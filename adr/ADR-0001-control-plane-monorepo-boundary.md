# ADR-0001: 控制面采用单仓多应用，而不是另起仓库或并入 rginx 主进程

状态：`Accepted`  
日期：`2026-04-16`

## 背景

`rginx` 当前已经具备以下可复用资产：

- 配置模型
- 配置校验与编译链
- reload / restart 边界
- 运行时快照
- 本地 admin socket 控制面

如果控制面另起仓库，后续很容易出现：

- 配置语义与节点实现漂移
- 版本兼容难以管理
- 发布节奏彼此制约

如果将控制面直接并入 `rginx` 主进程，则会引入：

- Web 框架和数据库依赖膨胀
- 数据面与控制面的故障域混合
- 节点进程职责失焦

## 决策

控制面采用“单仓多应用”的方式实现：

- 保持控制面与边缘节点在同一仓库
- 保持控制面与边缘节点为不同 crate、不同进程、不同容器
- 不将控制面代码并入 `rginx` 数据面主进程

控制面推荐分层如下：

- `rginx-control-api`
- `rginx-control-service`
- `rginx-control-store`
- `rginx-control-worker`
- `rginx-node-agent`
- `web/console`

## 后果

正面影响：

- 能直接复用 `rginx-core` 与 `rginx-config`
- 能保持控制面与数据面的版本语义一致
- 单仓协作成本较低
- 便于统一测试与发布

负面影响：

- 仓库规模会变大
- 前后端、数据面、控制面会共享同一个变更历史
- workspace 和 Docker 编排复杂度上升

## 不采纳方案

### 方案 A：另起控制面仓库

不采纳原因：

- 会导致配置编译能力双份实现或跨仓强耦合
- 版本兼容和联调成本更高

### 方案 B：把控制面直接并进 `rginx`

不采纳原因：

- 数据面和控制面的依赖、资源模型、故障域完全不同
- 会放大边缘节点进程复杂度
