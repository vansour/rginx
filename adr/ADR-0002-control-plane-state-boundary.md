# ADR-0002: Postgres 是真相源，Dragonfly 是加速层

状态：`Accepted`  
日期：`2026-04-16`

## 背景

控制面同时需要两类状态：

- 可审计、可恢复、可追溯的业务状态
- 高吞吐、低延迟、可过期的临时状态

两类状态的存储要求并不一致。

## 决策

采用双存储边界：

- `Postgres` 保存控制面的业务真相源
- `Dragonfly` 保存高频临时状态与异步任务加速数据

Postgres 必须保存：

- 用户、角色、权限
- 集群、节点、节点元数据
- 配置版本
- 发布任务与批次
- 审计日志

Dragonfly 负责：

- 登录会话
- 节点在线短缓存
- 任务队列
- 幂等键
- 部署锁
- SSE / WebSocket fanout
- 短期事件流

## 后果

正面影响：

- 业务真相可重建
- Dragonfly 的高性能能力可以被充分利用
- 队列、锁、实时广播能力不必全部压在数据库上

负面影响：

- 系统复杂度上升
- 需要明确双写和一致性边界
- 运维组件增多

## 设计约束

- 任何关键发布状态都必须可从 Postgres 还原
- Dragonfly 丢失时，系统允许失去短期缓存，但不允许丢失真实业务记录
- Worker 处理长任务时，必须先写数据库再更新 Dragonfly 的短期状态

## 不采纳方案

### 方案 A：只用 Postgres

不采纳原因：

- 实时广播、队列、分布式锁、短缓存会变得笨重
- API 和 Worker 的延迟会更高

### 方案 B：只用 Dragonfly

不采纳原因：

- 无法提供强审计、强追溯、强恢复能力
- 不适合作为控制面的唯一持久化系统
