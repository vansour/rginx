# ADR-0003: Node Agent 主动连接控制面并拉取任务

状态：`Accepted`  
日期：`2026-04-16`

## 背景

边缘节点普遍位于受限网络环境中，控制面直接反向连接节点存在以下问题：

- NAT / 防火墙环境复杂
- 控制面需要持有节点可达性假设
- 远程推送模型会增加网络与安全复杂度
- 直接 SSH 或远程执行不利于容器化部署

同时，`rginx` 当前的运行时控制接口本质上是本地 Unix socket。

## 决策

每个边缘节点部署一个 `rginx-node-agent`：

- agent 与本地 `rginx` 共享 `/run/rginx/admin.sock`
- agent 主动向控制面注册
- agent 定期发送 heartbeat
- agent 周期性拉取待执行任务
- agent 在本地写配置并触发 reload
- agent 回报执行结果和运行态快照

控制面不直接 SSH 到节点，也不把配置变更直接 push 到节点进程。

## 后果

正面影响：

- 更适配 Docker-only 部署
- 更适配边缘节点在复杂网络下的连接模型
- 控制面和节点之间的安全边界更清晰
- 控制面只需要暴露稳定的 API，不依赖节点反向可达

负面影响：

- agent 成为新增组件
- 发布延迟取决于拉取周期
- 任务调度与幂等需要明确设计

## 设计约束

- agent 只能操作本地 `rginx`，不操作其他节点
- agent 拉取任务必须支持幂等
- 控制面必须允许节点短时离线后恢复同步
- 节点状态和部署结果都必须回写 Postgres

## 不采纳方案

### 方案 A：控制面直接推送到节点

不采纳原因：

- 需要管理节点连通性与暴露面
- 会把网络复杂度和认证复杂度转移到控制面

### 方案 B：控制面通过 SSH 运维节点

不采纳原因：

- 不适合 Docker-only 和容器化路径
- 难以标准化幂等行为和可审计链路
